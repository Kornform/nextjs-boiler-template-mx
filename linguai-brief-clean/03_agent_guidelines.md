# Agent Guidelines

## Purpose

This file exists to prevent over-constraining the project too early.

## Rules

1. Treat this as a website project for a WordPress plugin, not as a dashboard or SaaS app.
2. Do not lock the visual direction prematurely.
3. Do not assume old design prompts are still valid.
4. Do not introduce detailed color systems, typography systems, or motion rules unless they are part of the current task.
5. Do not hardcode a homepage structure unless the current task explicitly asks for one.
6. Do not write final copy unless the task explicitly asks for copy.
7. Keep product positioning focused on clarity, usefulness, control, and reduced translation effort.
8. Translate technical features into user-facing benefits when working on content.
9. Keep claims realistic and verifiable.
10. Keep the project easy to rename, restyle, and restructure later.

## Preferred Workflow

When working on the project, the order should generally be:

1. understand the product and audience
2. define the page goal
3. define the information needed on the page
4. explore structure
5. explore design direction
6. implement with the approved stack

## Anti-Patterns

Avoid these unless explicitly requested:

- frozen design systems too early
- fixed color palettes from old briefs
- mandatory layout prescriptions
- overly technical hero messaging
- implementation detail overload in early design phases
- fake precision around pricing, performance, or claims

## Current Strategic Bias

The product itself is technically capable, but the website should not become difficult or rigid because of that.
